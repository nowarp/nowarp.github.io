TON Security Risks: A static analysis perspective

January 26, 2025 · 7 min read
Georgiy Komarov

Introduction​

Smart contracts are unforgiving. A single bug can vaporize millions of dollars. If you're coming from web development, forget everything you know about "move fast and break things" - here, breaking things means actually breaking things. With money. Real money.

This is where static analysis comes in. It's a technique that examines your code before deployment to automatically detect potential vulnerabilities. While no automated tool can guarantee perfect security, static analysis can identify many common security issues early in development, when they're cheapest to fix.

This article will:

  • Introduce fundamental concepts of static analysis for smart contract security
  • Demonstrate how static analysis tools fit into TON's security landscape
  • Present concrete capabilities and limitations of automated security analysis
  • Showcase practical applications through Misti
  • Give practical security advises

Understanding static program analysis enables you to add an additional layer of automated security verification to your development process, catching some vulnerabilities before they reach production.

Static Program Analysis 101​

Security tooling is an essential part of modern smart contract development, serving as the first line of defense against vulnerabilities. While manual code review remains crucial, automated analysis tools can systematically identify classes of bugs that would be tedious and error-prone to catch by hand.

Static program analysis operates by examining program code without actually executing it. The fundamental approach involves constructing abstract models of program behavior - typically through control flow graphs (CFGs) and data flow analysis. These models allow us to reason about program properties without running into the undecidability issues that would come with trying to analyze all possible runtime behaviors.

In essence, the approach is straightforward: we read and analyze the code structure to identify potential security vulnerabilities, without executing the code itself.

The widely used techniques that will be illustrated in this post include:

  • Control flow analysis: Mapping possible execution paths through the program
  • Data flow analysis: Tracking how values propagate through variables
  • Datalog-based analysis: Uses translation to Datalog in order to reason about desired security properties
  • Taint analysis: Following potentially dangerous input data through the program

Limitations of Static Analysis​

There are also different techniques of analysis exists, which could be used to explore concrete paths of execution, prove some properties on a limited domain of program, prove some properties of the program with more accuracy and so on, but the core issues are always the same: static undecidability. This fundamental limitation, proven by Rice's theorem, establishes that determining any non-trivial semantic property of programs is undecidable. In practical terms, this means that no static analysis tool, regardless of its sophistication, can guarantee the detection of all potential vulnerabilities or prove their absence. We can approximate, we can detect certain classes of issues, but complete verification remains mathematically impossible. This reality shapes how we approach smart contract security, emphasizing the importance of combining multiple analysis techniques with thorough manual review.

Static Analysis for Web3 and TON​

Smart contracts are just programs executing in blockchain. They have some differences, but fundamentally the same techniques of classic program analysis might be applied to smart contracts analysis.

Future Directions​

There are many possibilities in TON security automation.

Our concrete steps in Misti for the future:

  • Implementing IFDS with path-sensitivity tracking in order to improve accuracy of interprocdural taint analysis
  • Implement more Tact detectors using advanced static analysis techniques
  • Improve integrability and API to support third-party developers
  • Generic Datalog translation
  • Better tooling for auditors to actually understand the structure of contracts

Overall, Misti still following the development of the Tact language as the most reasonable language in the ecosystem which will become a standard in 2025. Also, there is an ongoing work on a tool capable to analyze both: legacy FunC code and Tact which will use a different approach from Misti. It will be announced later this year.

Conclusion​

We've considered the basic information about static program analysis and it's application to the TON security landscape. Now, after reading this, here are some concrete steps to increase security you should be thinking about:

  • Integrate security tools: Smart contracts handle real money and are unforgiving - a single bug can be catastrophic. Use every possibility to make your code secure. Static analysis is a powerful first line of defense, catching common vulnerabilities early and automatically. But it's just one piece of the security puzzle. Use every security tool you can, understand how it works and what can it actually do.
  • Apply development practices: testing, design, formal specification or at least documentation. This develops a separate post and might be highlighted in this blog later. Prefer a safe language.
  • Set up processes: security not only about analysis
  • Do security audit: While static analysis enhances the security process, it cannot replace thorough manual audits. For production contracts, professional security audits remain essential. You could find our contacts on this site or browse security teams collaborating with TF.
